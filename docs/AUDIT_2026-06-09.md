## 🔴 Критично / Высокий риск

### 1. Конкурентность стока (race condition, оверселл)
Сток проверяется через `SELECT`, затем уменьшается `stock_quantity -= qty` **без
блокировки строки** и **без CHECK ≥ 0**. Два параллельных заказа на последний
товар оба проходят проверку → оверселл и отрицательный сток.
- Доказательство: `back/app/services/orders.py:84` (проверка), `:143` (списание);
  у модели `Product` нет констрейнта `CHECK (stock_quantity >= 0)`.
- Решение: `SELECT … FOR UPDATE` на товарах внутри транзакции **или** атомарный
  `UPDATE … SET stock = stock - q WHERE stock >= q` + DB-констрейнт `stock ≥ 0` +
  тест на гонку.

### 2. Тестовая БД = боевая
pytest гоняет по той же БД, на которую указывают `POSTGRES_*`, и в teardown
**удаляет все строки** из 7 таблиц. Изолированной test-БД нет. Запуск тестов
против прод-БД сотрёт данные.
- Доказательство: `back/tests/conftest.py:24‑40` (общий `engine` + `delete(model)`).
- Решение: отдельная `POSTGRES_DB_TEST` / транзакционный rollback на тест /
  testcontainers.

### 3. Realtime не масштабируется
WS-менеджер хранит подключения **в памяти процесса**. `Dockerfile` запускает
`--workers 4` → новый заказ бродкастится только клиентам своего воркера. Сейчас
работает лишь потому, что compose переопределяет команду на 1 воркер.
- Доказательство: `back/app/services/realtime.py` (in-memory `list`) +
  `back/Dockerfile` CMD `--workers 4`.
- Решение: Redis pub/sub (или Postgres `LISTEN/NOTIFY`) как шина для WS-бродкаста.

### 4. Нет TLS / обратного прокси в проде
`back/compose.prod.yaml` поднимает только `db`+`backend` (без TLS, backend голым
`:8000`); **фронт-сервиса в prod-compose нет вообще**. HTTPS-терминатора нет.
- Доказательство: `back/compose.prod.yaml`.
- Решение: Traefik/Caddy/nginx с Let's Encrypt; единый prod-compose со всеми
  тремя сервисами за обратным прокси.

### 5. Нет CI/CD
Каталога `.github/workflows` нет. Гейты (lint/typecheck/build/pytest) не
автоматизированы — при verification-first это дыра.
- Решение: GitHub Actions — back (ruff/mypy/pytest на test-БД) + front
  (lint/typecheck/build/vitest) + e2e (Playwright).

## 🟡 Низкий риск / тех-долг

- **WS-токен в query-string** (`/admin/orders/ws?token=`) — может утечь в логи
  прокси/сервера. `admin.py:467`.
- **Нет code-splitting** — один чанк 530 КБ (vite warning). Нет `React.lazy` на
  admin/account-маршрутах. Большинство `<img>` без `loading="lazy"`.
- **Дублирование** `get_current_user` и `get_user_from_token` (`deps.py:34` и `:78`).
- **Auth-ошибки отдают 403** вместо 401 для невалидного токена (`deps.py:41`).
- **Bot-token Telegram в БД плейн-текстом** (`telegram_settings.bot_token`) —
  секрет в дампах БД (компромисс ради редактируемости).
- **Нет лимита на кол-во позиций/qty** в заказе — лёгкий DoS на расчёт quote.
- **Hard-delete каталога** без soft-delete/аудита; удаление категории с товарами
  может упасть на FK.
- **`prestart` гоняет миграции на каждом старте** — при нескольких репликах
  возможна гонка `alembic upgrade`.
- **Сознательно отложено в ТЗ** (не дефекты MVP): страница товара, email-
  уведомления покупателю, отмена заказа покупателем, CSV/Excel-импорт,
  настраиваемый порог low-stock. См. `docs/backend/product_spec.md`.
